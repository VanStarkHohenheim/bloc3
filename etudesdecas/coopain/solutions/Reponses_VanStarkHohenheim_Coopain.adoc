== MISSION 1 - Gestion des demandes d'insémination

=== Question 1.1 : Règle de gestion pour choisir une paillette

**Réponse :**

La requête montre que l’on sélectionne une paillette enrichie  
Elle doit correspondre au taureau demandé  
Le stock doit être disponible (au moins 1)  
On choisit la plus ancienne date de collecte  

Donc on utilise une logique FIFO : on prend le plus ancien lot

=== Question 1.3 : Requête SQL - Tarifs pour un taureau

**Réponse :**

[source,sql]
----
SELECT tp.libelle, t.prixPaillette
FROM Tarif t
JOIN TypePaillette tp ON t.idTypePaillette = tp.id
WHERE t.idNationalTaureau = 'FR0103015562'
ORDER BY tp.libelle;
----

=== Question 1.4 : Requête SQL - Liste des taureaux triés par stock

**Réponse :**

[source,sql]
----
SELECT 
    t.idNational,
    t.nom,
    SUM(l.stockRestant) AS stockTotal
FROM Taureau t
LEFT JOIN Lot l ON t.idNational = l.idNationalTaureau
GROUP BY t.idNational, t.nom
ORDER BY stockTotal ASC;
----

=== Question 1.5 : Schéma de données pour la gestion des demandes

**Réponse :**

On ajoute plusieurs entités pour gérer les demandes  

- Vache (identifiant, nom, lien avec adhérent)  
- DemandeInsemination (date, vache, taureau, etc.)  
- Inseminateur (infos + secteur + véhicule)  
- Secteur (zone géographique)  
- Vehicule (immat, modèle, date)  
- Adherent (client)  

Ces tables sont reliées entre elles pour suivre une demande complète

== MISSION 2 - MobiSemin Android

=== Question 2.1 : Scénarios d'erreur - Saisie de tournée

**Réponse :**

On ajoute des contrôles pour éviter les erreurs  

Si la date est dans le futur → erreur  
Si km fin < km début → erreur  
Si km invalide → erreur  
Si km = début → accepté (0 km)  
Si saisie trop tardive → avertissement

=== Question 2.3 : Analyse DICP - Risques en cas de perte/vol

**Réponse :**

Confidentialité :  
Les données peuvent être consultées (clients, tournées)  
La localisation peut être récupérée  

Intégrité :  
Les données peuvent être modifiées  
Risque de fraude ou erreurs  

Solutions : chiffrement, authentification, logs  

---

=== Question 2.4 : Vulnérabilité OWASP - Stockage du mot de passe

**Réponse :**

Le mot de passe ne doit pas être stocké en clair  
Sinon il peut être facilement récupéré  

Il faut utiliser :  
- un hash sécurisé (bcrypt, Argon2)  
- des tokens pour les sessions  
- HTTPS pour la transmission

public ArrayList<Adherent> getAdherents() {
    ArrayList<Adherent> adherentsUniques = new ArrayList<>();
    
    for(Visite v : laTournee.getLesVisites()) {
        Adherent adh = v.getLeAdherent();
        
        if(!adherentsUniques.contains(adh)) {
            adherentsUniques.add(adh);
        }
    }
    
    return adherentsUniques;
}

=== Question 3.3 : Attaque API - Synchronisation des tournées

**Réponse :**

Une attaque possible est le Man-in-the-Middle  

L’attaquant intercepte les données  
Il peut modifier les kilomètres ou horaires  
Il peut aussi injecter du SQL  

Solutions :  
- HTTPS obligatoire  
- validation côté serveur  
- requêtes préparées  
- signature des données
